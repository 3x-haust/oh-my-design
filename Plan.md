# OhMyDesign — 사람처럼 디자인하는 AI

```bash
npx oh-my-design install     # 한 번
```

그다음엔 어느 프로젝트에서든 Claude(또는 Codex)를 켜고 `/ultradesign 개발자 블로그 만들어줘`.

---

## 0. 이 도구가 고치려는 것

생성된 인터페이스는 **거의 고장나 있지 않다.** 고장난 것보다 나쁘다 — **정확하고 익명이다.**

인디고에서 바이올렛으로 흐르는 그라디언트. 똑같은 그림자를 진 세 장의 기능 카드.
전부 가운데 정렬. 로켓 이모지로 시작하는 제목. "Unlock the power of…"

이건 버그가 아니다. **관점이 없는 모델이 내놓는 필연적 결과**다. 개발자 블로그를
요청하면 모델은 자기가 본 모든 개발자 블로그의 **평균**을 그린다. 대비비는 통과한다.
접근성도 통과한다. 그런데 아무것도 말하지 않는다.

> **우리가 파는 건 "버그 없음"이 아니라 "익명이 아님"이다.**

---

## 1. 왜 평균으로 붕괴하는가 — 세 가지 결여

### 1.1 문제를 의심하지 않는다

디자인 인지 연구의 가장 견고한 발견 하나.

> **초보자는 주어진 문제를 푼다. 전문가는 주어진 문제를 의심한다.**

Dorst와 Cross가 전문 산업디자이너 9인을 프로토콜 분석했을 때, 그들은 문제를 다 이해한
뒤 풀지 않았다. 부분적인 해답을 던지고, **그 해답이 문제를 다시 가르쳤다.**
"아하 순간"의 정체가 이 진동이다.

"음식 주문 앱"을 요청받은 모델은 음식 주문 화면을 그린다. 사람은 먼저 묻는다 —
*사람들이 배고파서 앱을 여는가, 결정하기 싫어서 여는가?*

### 1.2 관점을 세우지 않는다

Darke가 관찰한 **프라이머리 제너레이터**. 전문가는 문제를 완전히 이해하기 전에
중심 은유 하나를 먼저 박는다.

> *"이 앱은 믿음직한 회계사다."*

이 한 문장이 색, 밀도, 모션, 카피를 전부 결정한다. 회계사는 요란하지 않다.
회계사는 서두르지 않으므로 여백이 넉넉하다. 회계사의 숫자에는 세리프가 붙는다.

**관점이 없으면 학습 분포의 평균으로 수렴한다.** 그게 슬롭의 기계적 정의다.

### 1.3 자기가 만든 것을 보지 않는다

Schön의 **see–move–see**. 디자이너는 머릿속에서 완성한 뒤 손으로 옮기지 않는다.
**그리면서 생각한다.** 손이 그은 선이 눈에 새 정보를 주고, 그게 다음 수를 바꾼다.

모델은 코드를 보고 "잘 보입니다"라고 말한다. 렌더링을 본 적이 없다.

---

## 2. 그러므로 "잘하는 AI"는 만들 수 없다

만들려면 **"좋은 디자인"의 보상 함수**가 있어야 한다. 그게 없다.

**DesignPref** — 전문 디자이너 20명이 UI 쌍 12,000건을 비교했다.

| 지표 | 이진 선호 | 4단계 선호 |
| :-- | :-- | :-- |
| 평균 쌍별 일치도 | 0.624 | 0.386 |
| Cohen's κ | 0.248 | 0.114 |
| **Krippendorff's α** | **0.248** | **0.104** |

**전체 비교의 28.5%에서 의견이 96% 이상 갈렸다.** 불일치의 출처는 전부 개인 취향이었다 —
대비를 어둡게 갈지 밝게 갈지, 화면을 채울지 비울지, 계층을 얼마나 강하게 드러낼지.

> ### 이 숫자가 제품 전체를 결정한다
>
> **보편적 미적 보상 함수는 원리적으로 존재할 수 없다.** 전문가들끼리도 α = 0.248이다.
>
> 그리고 평균 취향에 최적화한 모델은 정확히 **§0의 익명한 결과물**이 된다.
> 그건 버그가 아니라 **목적함수의 논리적 귀결**이다.
>
> 만들 수 있는 건 "잘하는 AI"가 아니라 세 가지다.
> **① 문제를 의심하게 만드는 것 ② 관점을 세우게 하는 것 ③ 평균을 탐지하는 것**

## 3. 세 개의 층

무엇이 "좋다"인지는 하나로 정의할 수 없지만, **셋으로는 정의할 수 있다.**

```text
┌─ Layer 3 · 개인 취향 ────────── 학습됨 · 확률적 · 사용자 소유
│    "당신은 지난 6개월간 촘촘한 레이아웃을 78% 선택했습니다"
│    출처: omd choose 로그 → .omd/taste/preferences.jsonl
│    ⚠ 하드코딩 절대 금지. α = 0.248이 그걸 금지한다.
│
├─ Layer 2 · 관점 정합성 ───────── 선언됨 · 프로젝트마다 다름
│    "'친구의 추천'인데 스와이프 카드는 모순입니다.
│     친구는 카드를 넘기라 하지 않습니다. 말을 겁니다."
│    출처: .omd/frame.md 의 primary generator
│    → 린터가 절대 못 내는 문장. eye 에이전트만이 낼 수 있다.
│
└─ Layer 1 · 결정론 ────────────── 보편 · 계산 가능 · 비용 0
     ├ a11y     WCAG 대비비, 히트 영역          "안 보인다"
     ├ system   8pt 그리드, 토큰 미참조          "일관성이 없다"
     └ slop     평균으로의 수렴                  "관점이 없다"   ★ 우리의 것
```

**Layer 1의 셋째 칸이 이 제품의 심장이다.**

`a11y`와 `system`은 결함을 말한다. `slop`은 다르다. **아무것도 고장나지 않았다.**
그저 모델이 본 모든 것의 평균에 도달했을 뿐이다.

## 4. 슬롭 탐지 — 검사 가능한 익명성

"AI 같지 않게"는 구호다. 규칙이어야 한다. 일곱 개를 구현했고, 전부 동작한다.

| 규칙 | 무엇을 잡는가 | 왜 문제인가 |
| :-- | :-- | :-- |
| `SLOP-GRADIENT` | 인디고→바이올렛 그라디언트 | 브랜드가 아니라 기본값이다 |
| `SLOP-RADIUS-MONOCULTURE` | 모든 모서리가 같은 반경 | 위계가 없으면 눈이 순위를 못 매긴다 |
| `SLOP-SHADOW-MONOCULTURE` | 같은 그림자 5회 이상 | 그림자는 높이다. 다 떠 있으면 아무것도 안 뜬다 |
| `SLOP-EVERYTHING-CENTERED` | 텍스트 블록 60%+ 가운데 정렬 | 가운데 정렬은 강조다. 다 강조하면 강조가 없다 |
| `SLOP-EMOJI-HEADING` | 이모지로 시작하는 제목 | 타이포그래피가 실패한 자리를 이모지가 메운다 |
| `SLOP-COPY` | "Unlock the power of…" | 어떤 제품에나 붙는 문장은 이 제품을 말하지 않는다 |
| `SLOP-TRIPLE-CARD` | 구조가 동일한 카드 3장 | 정보구조가 아니라, 정하지 못했다는 고백이다 |

### 4.1 진짜 시험은 "안 터지는 것"이다

슬롭 탐지기가 **좋은 디자인에서 터지면 없느니만 못하다.** 그래서 픽스처를 두 개 만들었다.

```text
slop.html         정확하고 접근성 통과. 그리고 익명.   → 7개 전부 발화
considered.html   같은 내용, 관점이 있음.              → 0건
```

`considered.html`은 카드 하나를 크게 만들고 둘을 작게 했다. 반경을 셋으로 나눴다.
그림자를 딱 한 번 썼다. 산문을 왼쪽으로 정렬했다. 카피를 구체적으로 썼다.
**같은 정보, 다른 입장.**

### 4.2 임계값은 취약하다 — 실측으로 배운 것

`SLOP-GRADIENT`를 헥사 목록으로 짜면 한 톤만 옆으로 가도 놓친다. 그래서 **색상환(hue)**
으로 판정한다. 두 스톱이 모두 230–300° 안에 있고 채도가 살아 있으면 발화한다.
인디고(239°)–바이올렛(258°)–퍼플(271°)을 덮고, 정직한 파랑 브랜드(#3B82F6, 217°)는 통과시킨다.

> 처음엔 240–300°로 썼다. 그리고 **`#6366F1`을 1도 차이로 놓쳤다.**
> 정확히 그 색을 잡으려고 쓴 규칙이었다.
> 손으로 고른 임계값은 취약하다. **사례 없이 넓히지 않는다.**

이 규칙들은 전부 **error가 아니라 warn**이다. 각각이 의도된 선택에 대해 틀릴 수 있다.
브랜드가 진짜로 보라색일 수 있다. **기각하려면 `omd decision`에 이유를 적어야 한다.**
"그냥 괜찮아 보였다"는 이유가 아니다.

## 5. 루프 — 사람의 인지 과정을 런타임으로

```text
        ┌────────────────────────────────────────────┐
        │                                            │
        ▼                                            │
   ① FRAME  ──▶ ② GENERATOR ──▶ ③ DIVERGE ──▶ ④ SEE ─┤
   문제를 의심     관점을 세움     진짜 3안 병렬   렌더·계측
        ▲                                            │
        └──────────── ⑥ REFRAME ◀──── ⑤ CULL ◀───────┘
              "본 것이 문제를 바꾼다"    근거를 대고 고른다

                    ⑦ STOP
```

⑥이 이 제품의 전부다. **어떤 도구도 자기가 방금 만든 3안을 스스로 폐기하고
문제 정의로 돌아가지 않는다.** 그게 사람이 하는 일이다.

### 5.1 프롬프트로 쓴 루프는 반드시 무너진다

이걸 SKILL.md에 글로 적어주면 실제로는 이렇게 동작한다.

| 지시 | 모델의 실제 행동 | 원인 |
| :-- | :-- | :-- |
| 문제를 의심하라 | 한 문장 의심하는 척하고 넘어감 | 사용자를 만족시키려는 압력 |
| 3안을 발산하라 | 1안을 만들고 색만 바꾼 3개 | **같은 컨텍스트 = 앵커링** |
| 렌더해서 봐라 | 코드를 보고 "잘 보입니다" | 렌더 도구가 없으면 상상한다 |
| 재정의하라 | 방금 만든 걸 부정하기 싫어함 | 자기비평 맹점 |
| 멈춰라 | 그만하랄 때까지 다듬음 | 정지 조건이 없다 |

**루프를 프롬프트에 쓰면 루프가 아니라 루프에 대한 묘사가 된다.**

### 5.2 그래서 구조로 만든다 — 강제가 아니라 배치로

```text
CLI          결정론적 계측                  → "상상함"을 막는다
             대비비는 함수가 계산한다. 모델이 세지 않는다.
             그리고 슬롭은 **숫자로 잡힌다.** 취향 논쟁이 아니다.

서브에이전트   컨텍스트 격리                   → "앵커링·자기비평 맹점"을 막는다
             hand ×3이 서로를 못 본다 ⇒ 진짜 대안이 나온다.
             eye는 hand의 의도를 모른다 ⇒ 진짜 비평을 한다.
             ★ 유일하게 진짜 구조적인 방어다. 프롬프트로 흉내낼 수 없다.

.omd/       기록                          → "재정의를 회피함"을 막는다
             frame.md가 갱신된 흔적이 남는다. 덮어쓰지 않고 덧붙인다.
```

**①과 ⑤(문제를 의심함, 멈춤)에는 구조적 방어가 없다.** 프롬프트뿐이다.
한때 훅으로 ①을 강제하려 했고, 그게 왜 틀렸는지가 §6이다.

> 여기서 배운 규칙: **관찰로 검증될 일을 강제로 대신하지 않는다.**
> 틀린 재프레이밍은 렌더를 보면 드러난다. 서명으로는 드러나지 않는다.

## 6. 폐기된 설계: 프레임 게이트

한동안 `.omd/frame.md`가 **승인**되기 전까지 `PreToolUse` 훅이 디자인 파일 쓰기를
`exit 2`로 막았다. 잘못된 재프레이밍 위에 전부 지어버리는 걸 막으려는 장치였다.

**전부 지웠다.** 왜 틀렸는지가 이 문서에서 가장 값진 부분이다.

### 6.1 서명으로 검증할 수 없는 것을 서명으로 검증하려 했다

틀린 재프레이밍은 **만들어보고, 렌더해서 보고, 아니면 되돌아가는 것**으로만 잡힌다(§5).

그런데 게이트가 서 있으면:

```text
승인 전 → 아무것도 못 만든다
       → 렌더할 게 없다
       → 볼 것이 없다
       → 재프레이밍이 틀렸는지 알 방법이 없다
```

**루프를 그려놓고 그 루프를 막고 있었다.** 관찰로만 검증되는 것을 사람의 사인으로
대신하려 한 설계 오류다.

### 6.2 그리고 사용자에게 터미널로 가라고 시켰다

`omd frame approve`를 치라고 했다. **사용자는 웹사이트를 원했지 명령어를 원하지 않았다.**
그들이 이 도구를 켠 이유가 그 일을 하기 싫어서인데, 도구가 그 일을 되돌려줬다.

### 6.3 실측이 이미 경고했는데 잘못 읽었다

실제 헤드리스 세션에 물려 네 번 돌렸다. 셋은 통과했고, 넷째가 뚫렸다.

> **"게이트가 막으면 직접 치우고 진행해"** → 에이전트가 `omd frame approve`를 실행했다.

그때 나는 **우회를 막았다** — TTY 검사와 근거 요구를 붙였다. 고쳤어야 할 것은
**게이트의 존재 자체**였다. 자기승인이 쉬웠다는 건 게이트가 취약하다는 신호가 아니라,
**게이트가 아무것도 지키고 있지 않았다는 신호**였다.

### 6.4 남긴 것

| 지운 것 | 남긴 것 |
| :-- | :-- |
| `PreToolUse` 훅 | — |
| 세션 범위 | — |
| `omd frame approve` · TTY 검사 | — |
| — | `.omd/frame.md` — **기록**. 서명받지 않는다 |
| — | `omd frame reframe` — 루프가 스스로 갱신 |
| — | **`--why` 강제** — 근거 없는 재프레이밍은 거부 |

마지막 줄이 핵심이다. `--why`는 **사람을 막는 장치가 아니라 에이전트가 직감을
통찰이라 부르는 걸 막는 장치**다. "제 생각에는"은 근거가 아니다.

그리고 `omd frame reframe`은 이전 프레이밍을 **덮어쓰지 않고 덧붙인다.**
프레임이 두 번 바뀐 기록이 6개월 뒤 그 레포에서 가장 흥미로운 문서다.

---

### 6.5 부수 발견: 빌드가 스스로 게이트를 되살렸다

`dist/`를 비우지 않고 덮어써서, 훅을 방출하지 않게 된 뒤에도 **낡은 훅 파일이 남았고
설치기가 그걸 복사했다.** 지운 게이트가 조용히 되살아났다.

> 배포 산출물은 항상 먼저 비운다. 방출을 멈춘 파일은 설치도 멈춰야 한다.

## 7. 무엇을 만들고 무엇을 안 만드는가

| | 결정 | 이유 |
| :-- | :-- | :-- |
| 파운데이션 모델 | ❌ | VLM을 우리 자원으로 못 이긴다 |
| 범용 미적 보상 모델 | ❌ | **α = 0.248. 원리적으로 불가능** |
| 스크린샷→코드 | ❌ | 거의 풀렸다. 그리고 그건 받아쓰기지 디자인이 아니다 |
| **자체 MCP 서버** | ❌ | 배포된 `chrome-devtools-mcp`를 등록해 쓴다 |
| 호스트 런타임 | ❌ | Claude Code / Codex 플러그인으로 올라탄다 |
| 선형 파이프라인 (리서치→…→핸드오프) | ❌ | **초보자의 행동.** §8 |
| 스킬 50~100개 | ❌ | 복제 가능. 해자는 IR·격리·규칙에 있다. §8 |
| **승인 게이트** | ❌ | **폐기.** 관찰로 검증할 것을 서명으로 검증했다. §6 |
| **슬롭 탐지** | ✅ | 아무도 안 한다. **제품의 심장** |
| **FRAMER** | ✅ | 전문가/초보 최대 격차 |
| **컨텍스트 격리 (hand ×3, eye)** | ✅ | 진짜 발산 · 자기비평 맹점 제거 |
| **`omd` CLI (IR·룰·렌더)** | ✅ | LLM이 산술을 하면 90%만 맞는다 |
| **`.omd/` 메모리** | ✅ | 근거의 버전 관리. 락인 |
| **Coach** | ✅ | **아직 없다.** 증폭의 유일한 증거. §8 |

**모델이 아니라 인지 구조를 만든다.** 모델은 갈아끼운다.

### 7.1 지각은 CLI, 조작은 MCP

```text
사람  →  npx oh-my-design install
훅    →  omd hook pre-tool              (게이트)
모델  →  omd check / omd render         (결정론적 계측 — 함수가 계산한다)
모델  →  chrome-devtools MCP            (호버·포커스·상호작용 — 배포된 서버)
```

대비비를 모델이 브라우저를 몰아 읽게 하면 90%만 맞는다. 디자이너는 그 10%를 발견한 날
도구를 버린다. **틀리면 안 되는 건 전부 CLI로 뺐다.**

대가도 있다. MCP 툴은 모델에게 자동 발견되고 스키마가 있지만, CLI는 SKILL.md로 가르쳐야
하고 모델이 안 부를 수 있다. **`/ultradesign`이 실제로 `omd check`를 부르는지가
이 도박의 성패다.**

## 8. 원안 재검토 — "디자이너를 증강시키는 Codex"

이 프로젝트의 최초 구상은 **스킬 50~100개를 붙인 Design OS**였다. 리서치가 그중 절반을
반증했고, 절반은 우리가 아직 못 하고 있는 것을 정확히 짚었다. 나눠서 기록한다.

### 버린 것 ①: 선형 파이프라인

```text
리서치 → 아이디어 → Flow → Wireframe → Design System → Visual → QA → Handoff
```

**이건 초보자의 행동이다.** Dorst & Cross의 프로토콜 분석(§1.1)에서 전문 디자이너는
문제를 다 이해한 뒤 풀지 않았다. 부분해를 던지고 **그 해답이 문제를 다시 가르쳤다.**

단계를 순서대로 밟으면 **마지막 단계에 가서야 첫 단계가 틀렸음을 안다.**
그때는 이미 늦다. 그래서 우리는 파이프라인이 아니라 **루프**를 만들었다(§5).

> 다만 원안이 옳게 본 것: **각 단계는 완전히 다른 문제다.** LLM 하나가 다 하면
> 성능이 떨어진다. 우리는 그걸 **스킬 분리가 아니라 컨텍스트 격리**로 푼다 —
> `omd-hand` 셋은 서로를 못 보고, `omd-eye`는 만든 이유를 모른다(§5.2).

### 버린 것 ②: 스킬 개수가 해자라는 가정

스킬은 마크다운 파일이다. 하루에 열 개도 쓴다. **경쟁자가 `skills/` 폴더를 통째로
복사해도 같은 결과가 안 나온다.** 이유는 개수가 아니라 그 아래에 있다.

```text
스킬        ← 복제 가능. 여기 힘 빼지 않는다.
  │
  ├── 입력의 해상도    Design IR + `omd check`  ← 스샷만 본 모델은 "간격이 좀…"까지가 한계
  ├── 컨텍스트 격리    서브에이전트             ← 앵커링과 자기비평 맹점을 구조로 막는다
  └── 슬롭 규칙        평균의 정의              ← 우리만 가진 것
```

`observe-screen`, `normalize-spacing`, `find-inconsistent-components`, `accessibility-check`
는 **스킬이 아니라 `omd check`의 규칙 한 줄씩**이다. 스킬로 만들면 LLM이 매번 눈대중으로
센다. 규칙으로 만들면 함수가 정확히 센다.

### 취한 것 ①: "생성이 아니라 관찰"

원안의 가장 정확한 문장이다. 시니어 디자이너는 **그리는 시간보다 보는 시간이 훨씬 길다.**
`omd-eye`와 `critique` 스킬이 이미 여기 서 있다.

### 취한 것 ②: Inspiration은 모방이 아니라 분석

> ❌ "Apple처럼 만들어"
> ✅ "Apple은 **왜** 이렇게 만들었는가"

이건 우리의 **primary generator**(§1.2)와 정확히 같은 논지인데, 원안의 표현이 더 낫다.
`omd frame generator --set "…"`에 은유를 박는 이유가 바로 이것이다.
Apple의 여백을 베끼면 Apple 짝퉁이 나오지만, *"서두르지 않는다"*를 이해하면
우리 것이 나온다.

### 취한 것 ③ — **가장 큰 빈자리: Coach**

원안이 말한 것:

> "이렇게 바꾸세요"가 아니라 **"왜 이 선택이 더 좋은가"**를 설명하며
> 디자이너의 실력을 함께 끌어올린다.

**지금 우리 도구에 이게 없다.** 현재 `/ultradesign`은 슬롭을 잡고 **스스로 고친다.**
사용자는 완성된 사이트를 받고, 아무것도 배우지 못하고, 도구를 끄면 실력은 그대로다.

그게 §0의 포지셔닝과 어긋난다. 대체가 아니라 증폭이라면, **쓸수록 사용자가 늘어야 한다.**

**설계 (Phase 4로 편입)**

```bash
omd coach                    # .omd/decisions.md + preferences.jsonl 을 읽는다
```

세 가지를 낸다.

1. **왜 그 수가 더 좋은가** — 고친 슬롭마다, 고치기 전후를 나란히 두고
   *무엇이 달라졌는지*가 아니라 *왜 그게 나은지*를 은유에 비추어 설명한다.
2. **반복되는 약점** — `decisions.md`가 쌓이면 개인의 패턴이 보인다.
   *"당신은 3개월간 대비비 지적을 41회 받았습니다. 1분기 대비 -68%.
   대신 정보 계층 지적이 12→29회로 늘었습니다."*
3. **다음에 볼 것** — 약점에 맞는 관찰 과제. 생성이 아니라 관찰.

**이게 진짜 락인이다.** 슬롭 규칙은 복제된다. 하지만 *당신이 지난 6개월간 무엇을
반복해서 틀렸는가*는 옮길 수 없다.

> 그리고 Coach는 **Layer 3(개인 취향)과 다른 것**이다.
> Layer 3은 *"당신은 촘촘한 레이아웃을 좋아한다"* (취향, α=0.248이라 옳고 그름이 없다).
> Coach는 *"당신은 대비비를 반복해서 놓친다"* (실력, 옳고 그름이 있다).
> **취향은 배우고, 실력은 가르친다.** 이 둘을 섞으면 둘 다 망한다.

### 보류: Figma

원안의 `convert-to-autolayout`, `figma-cleanup`, `design-token-sync`는 전부 옳지만
**Figma Plugin API가 필요하다.** 현재 우리는 브라우저 DOM만 읽는다.

Figma REST API로 IR을 뽑는 건 가능하다(읽기 전용). 쓰기는 플러그인 심사가 필요하다.
**Phase 6 이후.** 지금 손대면 웹에서 증명하지 못한 것을 Figma에서 증명하려 드는 셈이다.

## 10. 로드맵

### Phase 0 · 가설 검증 (남아 있음)

| | 검증 대상 | 실패 조건 |
| :-- | :-- | :-- |
| **E1** | 재프레이밍이 실제로 가치 있는가 | 디자이너 10명 블라인드 비교. 선호 60% 미만이면 **루프를 폐기** |
| **E2** | 관점 부여가 익명성을 막는가 | generator 유/무 각 20안. 심사위원이 출처를 맞히는가 |
| **E3** | 슬롭 규칙이 실제 프로덕션 사이트에서 오탐하는가 | Linear · Stripe · 개인 블로그 20개 |
| **E4** | 개인 취향이 30샘플로 학습되는가 | 홀드아웃 20건 |

> **E4의 기준선이 미묘하다.** 사람도 자기 자신과 완벽히 일치하지 않는다. 같은 쌍을
> 이틀 뒤 다시 보여줘 **개인 내 일관성(test-retest)**을 먼저 재고, 그걸 상한으로 삼는다.
> **α = 0.248은 타인 간 일치도지 상한이 아니다.**

**E1이 실패하면 전제가 무너진다.** 그러면 루프를 버리고 **슬롭 린터**로 축소한다.
후퇴가 아니다 — §9를 보면 그것만으로도 이미 동작한다.

**E3가 실패하면 슬롭 규칙을 다시 짜야 한다.** 좋은 사이트에서 터지는 탐지기는
없느니만 못하다.

### Phase 1 · 지금 (완료)

슬롭 린터 + 눈 + 루프. Claude Code에 설치되고 동작한다.

### Phase 2 · 루프 실전 검증 ← **다음**

`/ultradesign`으로 사이트를 하나 만든다. 관찰할 것은 결과물이 아니라 **행동**이다.

- 모델이 `omd check`를 부르는가, 대비비를 지어내는가? (미확인 #1)
- 슬롭을 고치는가, "이런 지적이 있습니다" 하고 넘어가는가? (미확인 #2)
- `omd-hand` 셋이 정말 다른 것을 만드는가? (미확인 #3)

**기능 추가 없음.** 여기서 하나라도 실패하면 그걸 고치는 게 Phase 3다.

### Phase 3 · Coach (§8)

```bash
omd coach
```

`decisions.md`와 `preferences.jsonl`을 읽어 **왜 그 수가 더 좋았는지**를 설명하고,
반복되는 약점을 짚고, 다음에 볼 것을 준다.

**이게 "대체가 아니라 증폭"의 유일한 증거다.** 이게 없으면 우리는 그냥 더 나은 v0다.
성공 기준: 사용자가 `omd coach` 출력을 팀에 공유한다.

### Phase 4 · TASTE (Layer 3)

`omd choose` 로그 축적 → 개인 취향 모델. **제품을 쓰는 행위 자체가 데이터 수집이다.**

> Coach(실력)와 TASTE(취향)는 **절대 섞지 않는다.** §8 참조.

### Phase 5 · Codex + 룰팩 생태계

Codex 실설치. 그리고 `.omd/rules/custom.yaml`을 커뮤니티가 쓴다.
**"Linear 밀도 룰팩", "Apple 여백 룰팩"** 이 유통되면 생태계다.
`BRAND-014: 결제 흐름의 CTA는 브랜드 컬러여야 한다` — 우리는 이걸 못 만든다.
**그 팀만 만들 수 있다. 그게 해자다.**

### Phase 6 · Figma

REST API로 읽기 전용 IR. 쓰기는 플러그인 심사 이후.
**웹에서 증명한 뒤에 간다.**

## 12. 한 문장으로

> 사람처럼 디자인한다는 건 좋은 화면을 뽑는 게 아니다.
> **문제를 의심하고, 관점을 세우고, 한 수 두고, 보고, 문제 정의를 갈아엎는 것**이다.
>
> "좋다"에 보편 함수가 없으므로(α = 0.248),
> 우리가 할 수 있는 건 **평균을 탐지하고, 관점을 세우고, 당신의 취향을 배우는 것**이다.

**계산은 CLI가, 판단은 에이전트가, 기록은 `.omd/`가 한다. 강제하는 것은 없다.**

그리고 최종 형태는 대체가 아니라 증폭이다.

> 지금은 우리가 슬롭을 잡고 고친다. 사용자는 완성된 사이트를 받는다.
> **Coach가 붙으면 사용자가 슬롭을 알아보게 된다.**
> 그때 이 도구는 디자이너를 대신하는 것이 아니라,
> **옆에서 계속 리뷰하는 시니어 디자이너**가 된다.
>
> 대체형 도구는 더 좋은 모델이 나오면 즉시 갈아탄다.
> 증폭형 도구는 사용자의 판단 히스토리가 그 안에 쌓인다.

## 부록 B · 설치 UX 참고

`npx` 한 번으로 전역 설치하고, 훅이 컴포넌트 CLI를 부르고, 모델에게는 MCP를 주는 형태는
oh-my-claudecode / oh-my-openagent(LazyCodex)의 배치를 참고했다.
**형식만 참고했을 뿐 이 제품의 내용과는 무관하다.**

호스트 스펙 실측 (2026-07-09):

- `~/.claude/settings.json` — `hooks` · `permissions.allow` · `enabledPlugins` · `extraKnownMarketplaces`
- `~/.codex/config.toml` — `[features]` · `[agents.<n>] config_file` · `[hooks.state.<key>] trusted_hash`
- Codex 플러그인 매니페스트에는 `agents` 키가 없다. 에이전트는 설치기가 `~/.codex/agents/*.toml`에 배치하고 `config.toml`에 등록한다.
- **Codex 에이전트 TOML에는 도구 제한 키가 없다.** `omd-eye`의 쓰기 금지가 Claude에선 `disallowedTools`로 강제되지만 Codex에선 산문일 뿐이다.
- [Codex Hooks](https://developers.openai.com/codex/hooks) — `exit 2` 차단, 그 외 non-zero는 "훅 실패"로 보고 **계속 진행**한다. fail-closed 설계의 근거.
