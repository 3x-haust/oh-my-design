# OhMyDesign

> **컨셉을 잡고, 레퍼런스를 모으고, 원리를 뽑아, 내 문제에 맞게 조립한다.**
> 사람이 디자인하는 방식이다.

```bash
npx oh-my-design install
```

그다음엔 어느 프로젝트에서든 `/ultradesign 개발자 블로그 만들어줘`.
사용자는 아무것도 더 하지 않는다.

---

## 1. 고치려는 것

생성된 인터페이스는 **거의 고장나 있지 않다.** 고장난 것보다 나쁘다 — **정확하고 익명이다.**

인디고에서 바이올렛으로 흐르는 그라디언트. 똑같은 그림자를 진 세 장의 기능 카드.
전부 가운데 정렬. 로켓 이모지로 시작하는 제목. "Unlock the power of…"

이건 버그가 아니다. **관점이 없는 모델이 내놓는 필연적 결과**다. 개발자 블로그를
요청하면 모델은 자기가 본 모든 개발자 블로그의 **평균**을 그린다.
대비비는 통과한다. 접근성도 통과한다. 그런데 아무것도 말하지 않는다.

> **우리가 파는 건 "버그 없음"이 아니라 "익명이 아님"이다.**

---

## 2. 사람은 어떻게 디자인하는가 — 네 가지

모델에게 없는 것 넷. 이 넷이 제품의 골격이다.

### 2.1 문제를 의심한다

> **초보자는 주어진 문제를 푼다. 전문가는 주어진 문제를 의심한다.**

Dorst와 Cross가 전문 산업디자이너 9인을 프로토콜 분석했을 때, 그들은 문제를 다 이해한
뒤 풀지 않았다. 부분해를 던지고 **그 해답이 문제를 다시 가르쳤다.**

"음식 주문 앱"을 요청받은 모델은 음식 주문 화면을 그린다.
사람은 먼저 묻는다 — *배고파서 앱을 여는가, 결정하기 싫어서 여는가?*

### 2.2 컨셉을 먼저 박는다

Darke의 **프라이머리 제너레이터**. 전문가는 문제를 완전히 이해하기 전에
중심 은유 하나를 먼저 정한다.

> *"이 앱은 믿음직한 회계사다."*

이 한 문장이 색, 밀도, 모서리, 모션, 카피를 전부 결정한다.
회계사는 요란하지 않다. 서두르지 않으므로 여백이 넉넉하다.

**컨셉이 없으면 학습 분포의 평균으로 수렴한다.** 그게 슬롭의 기계적 정의다.

### 2.3 레퍼런스를 모은다 ★

**이 프로젝트가 가장 오래 놓치고 있던 것.**

사람은 컨셉을 정하고 나면 백지에서 그리지 않는다. **찾아본다.**

```text
컨셉: "조용한 서재"
   ↓
그럼 검색바는 어때야 하지?          → Linear, Raycast, Arc 를 연다
그럼 목록은 어떻게 밀도를 잡지?      → Notion, Bear, Things 를 연다
그럼 여백은 얼마나 줘야 하지?        → Stripe Docs 를 연다
   ↓
각각에서 **왜 그렇게 했는지**를 읽는다
   ↓
내 문제에 맞게 **조립한다**
```

레퍼런스는 베끼는 대상이 아니라 **관찰 대상**이다.
그리고 조립은 짜깁기가 아니라 **컨셉이라는 기준으로 걸러낸 선택**이다.

> *"Apple처럼 만들어"* 가 아니다.
> *"Apple은 **왜** 이렇게 만들었는가"* 다.

### 2.4 자기가 만든 것을 본다

Schön의 **see–move–see**. 디자이너는 머릿속에서 완성한 뒤 손으로 옮기지 않는다.
**그리면서 생각한다.** 손이 그은 선이 눈에 새 정보를 주고, 그게 다음 수를 바꾼다.

모델은 코드를 보고 "잘 보입니다"라고 말한다. 렌더링을 본 적이 없다.

---

## 3. 그러므로 "잘하는 AI"는 만들 수 없다

만들려면 **"좋은 디자인"의 보상 함수**가 있어야 한다. 그게 없다.

**DesignPref** — 전문 디자이너 20명이 UI 쌍 12,000건을 비교했다.

| 지표 | 이진 선호 | 4단계 선호 |
| :-- | :-- | :-- |
| 평균 쌍별 일치도 | 0.624 | 0.386 |
| Cohen's κ | 0.248 | 0.114 |
| **Krippendorff's α** | **0.248** | **0.104** |

**전체 비교의 28.5%에서 의견이 96% 이상 갈렸다.** 불일치의 출처는 전부 개인 취향이었다.

> **보편적 미적 보상 함수는 원리적으로 존재할 수 없다.**
> 평균 취향에 최적화한 모델은 정확히 §1의 익명한 결과물이 된다.
> **버그가 아니라 목적함수의 논리적 귀결이다.**
>
> 그러므로 우리가 만드는 것은 "좋은 디자인을 아는 AI"가 아니다.
> **문제를 의심하고, 컨셉을 세우고, 실물을 관찰해 원리를 뽑고, 자기가 만든 걸 보는 AI**다.

## 4. 세 개의 층

"좋다"를 하나로 정의할 수 없지만 **셋으로는 정의할 수 있다.**

```text
┌─ Layer 3 · 개인 취향 ────────── 학습됨 · 확률적 · 사용자 소유
│    "당신은 6개월간 촘촘한 레이아웃을 78% 선택했습니다"
│    ⚠ 하드코딩 절대 금지. α = 0.248이 그걸 금지한다.
│
├─ Layer 2 · 컨셉 정합성 ───────── 선언됨 · 프로젝트마다 다름
│    "'조용한 서재'인데 이 바운스 애니메이션은 모순입니다."
│    출처: .omd/frame.md 의 컨셉 + .omd/refs/ 의 원리
│    → 린터가 절대 못 내는 문장.
│
└─ Layer 1 · 결정론 ────────────── 보편 · 계산 가능 · 비용 0
     ├ a11y     WCAG 대비비, 히트 영역        "안 보인다"
     ├ system   8pt 그리드, 토큰 미참조        "일관성이 없다"
     └ slop     평균으로의 수렴                "관점이 없다"   ★
```

**Layer 1의 셋째 칸이 이 제품의 심장이다.**
`a11y`와 `system`은 결함을 말한다. `slop`은 다르다 — **아무것도 고장나지 않았다.**

## 5. 슬롭 탐지 — 검사 가능한 익명성

"AI 같지 않게"는 구호다. 규칙이어야 한다. 일곱 개, 전부 동작한다.

| 규칙 | 무엇을 잡는가 | 왜 문제인가 |
| :-- | :-- | :-- |
| `SLOP-GRADIENT` | 인디고→바이올렛 (hue 230–300°) | 브랜드가 아니라 기본값 |
| `SLOP-RADIUS-MONOCULTURE` | 모든 모서리가 같은 반경 | 위계가 없으면 눈이 순위를 못 매긴다 |
| `SLOP-SHADOW-MONOCULTURE` | 같은 그림자 5회+ | 그림자는 높이다. 다 떠 있으면 안 뜬 것 |
| `SLOP-EVERYTHING-CENTERED` | 텍스트 60%+ 가운데 정렬 | 가운데 정렬은 강조다. 다 강조하면 강조가 없다 |
| `SLOP-EMOJI-HEADING` | 이모지로 시작하는 제목 | 타이포그래피가 실패한 자리를 이모지가 메운다 |
| `SLOP-COPY` | "Unlock the power of…" | 어떤 제품에나 붙는 문장은 이 제품을 말하지 않는다 |
| `SLOP-TRIPLE-CARD` | 구조 동일 카드 3장 | 정보구조가 아니라, 정하지 못했다는 고백 |

### 5.1 진짜 시험은 "안 터지는 것"이다

좋은 디자인에서 터지는 탐지기는 **없느니만 못하다.** 픽스처를 두 개 만들었다.

```text
slop.html         정확하고 접근성 통과. 그리고 익명.   → 7개 전부 발화
considered.html   같은 내용, 관점이 있음.              → 0건
```

**실사이트로도 확인했다.** Linear를 우리 추출기로 읽으면:

```text
반경 사다리    4 / 6 / 8 / 12 / 16 / 50 / 9999px   ← 7단. 모노컬처 아님
높이(elevation) 3층                                 ← 그림자 5종 중 2종은 헤어라인
가운데 정렬    9%                                   ← 임계값 60%
토큰 커버리지  71%
```

> **실측이 내 문장을 하나 고쳤다.** 처음엔 "그림자 5종 = 높이 5층"이라고 썼다.
> 실제로는 그중 넷이 `0 0 0 1px` — **테두리를 그림자로 그린 것**이다.
> 높이는 3층이다. `elevationLevels`는 헤어라인을 세지 않는다.

**슬롭 규칙이 Linear를 슬롭이라 부르지 않는다.** 그게 이 규칙들이 살아남는 조건이다.

### 5.2 임계값은 취약하다

`SLOP-GRADIENT`를 헥사 목록으로 짜면 한 톤만 옆으로 가도 놓친다.
그래서 **색상환(hue)**으로 판정한다.

> 처음엔 240–300°로 썼다. 그리고 **`#6366F1`(239°)을 1도 차이로 놓쳤다.**
> 정확히 그 색을 잡으려고 쓴 규칙이었다.
> **손으로 고른 임계값은 취약하다. 사례 없이 넓히지 않는다.**

전부 `error`가 아니라 `warn`이다. 브랜드가 진짜로 보라색일 수 있다.
**기각하려면 `omd decision`에 이유를 적어야 한다.** "그냥 괜찮아 보였다"는 이유가 아니다.

## 6. 레퍼런스 — 제품의 두 번째 기둥

슬롭 탐지는 **평균을 지운다.** 그런데 지운 자리를 무엇으로 채우는가?
아무것도 안 하면 모델은 다시 학습 분포 안에서 고른다. **레퍼런스가 그 답이다.**

### 6.1 치명적 함정: Design Fixation

**Jansson & Smith (1991).** 아이디어 생성 전에 예시 그림 하나를 보여줬다.
결과는 네 개의 서로 다른 설계 과제에서 재현됐다.

> 예시의 **결함을 명시적으로 지적해준 뒤에도**,
> 피험자들의 설계에는 그 예시의 특징이 비정상적인 비율로 남았다.

즉 **레퍼런스를 순진하게 물어다 주면 슬롭이 줄지 않고 늘어난다.**
"Linear처럼 만들어줘"는 Linear 짝퉁을 낳는다. 그건 익명성의 다른 이름일 뿐이다.

후속 연구는 조건을 더 좁힌다 — **물리적/구체적 예시일수록 고착이 심하다.**
그리고 LLM은 시각 예시에 인간보다 더 강하게 고착될 이유가 있다:
그게 학습 목표 자체였다.

### 6.2 그래서: 픽셀이 아니라 측정값을 넘긴다

**모델에게 스크린샷을 주지 않는다. 숫자를 준다.**

우리 IR 추출기는 이미 라이브 프로덕션 사이트를 읽는다. Linear에서 **892노드, 371토큰.**
거기서 뽑는 것은 그림이 아니라 **불변량(invariants)**이다.

```text
omd ref add https://linear.app --as search-bar
```

```jsonc
// .omd/refs/linear.search-bar.json
{
  "source": "https://linear.app",
  "component": "search-bar",
  "invariants": {
    "spacingLadder":   [1,2,4,6,8,10,12,16,20,24,28,40,48],  // 사용량 90%를 덮는 값들
    "radiusLadder":    [4,6,8,12,16,50,9999],                // 7단. 물성이 다름을 뜻한다
    "elevationLevels": 3,                                     // 헤어라인 제외한 진짜 높이
    "centeredRatio":   0.0929,
    "tokenCoverage":   0.7068,
    "paddingWeight":   10.47
  },
  "principles": [                              // ← 여기가 사람이 읽는 부분
    "반경이 7단으로 갈린다. 입력·카드·아바타가 서로 다른 물성을 갖는다.",
    "가운데 정렬은 9%뿐. 강조로만 쓴다.",
    "테두리를 그림자로 그린다. 진짜 높이는 3층뿐이고, 나머지는 1px 선이다."
  ]
}
```

**`principles`는 VLM이 렌더 이미지와 불변량을 함께 보고 쓴다.**
*"어떻게 생겼는가"*가 아니라 ***"왜 그렇게 했는가"***를 쓴다. §2.3의 그 문장이다.

### 6.3 조립 — 컨셉이 필터다

레퍼런스 세 개를 모았다고 셋을 다 쓰지 않는다. **컨셉이 걸러낸다.**

```text
컨셉: "조용한 서재"

Linear    반경 5단        → 채택. 서재의 물건들은 서로 다른 재질이다.
Raycast   즉각적 모션      → 기각. 서재는 서두르지 않는다.
Stripe    넓은 여백        → 채택. 여백이 조용함이다.
Notion    촘촘한 밀도      → 기각. Stripe와 모순. 컨셉이 Stripe를 고른다.
```

**기각도 기록된다.** `omd decision "Raycast의 모션 기각" --why "서재는 서두르지 않는다"`

이게 짜깁기와 디자인의 차이다. 짜깁기는 좋아 보이는 걸 다 넣는다.
디자인은 **컨셉이라는 하나의 기준으로 대부분을 버린다.**

### 6.4 고착을 측정한다

레퍼런스를 썼는데 결과가 어느 한 레퍼런스와 너무 닮았다면, 그건 조립이 아니라 복제다.

```bash
omd ref distance <page>
```

산출물의 불변량 벡터를 각 레퍼런스와 비교한다.

```text
Linear   0.71   ⚠ 너무 가깝다. 반경 사다리·간격 스케일이 거의 동일하다.
Stripe   0.38
Notion   0.22
```

> **어느 레퍼런스와도 0.6 이상 가까우면 경고한다.**
> 여러 레퍼런스에서 원리를 뽑았다면, 결과는 그 어느 것과도 닮지 않아야 한다.

이게 Jansson & Smith에 대한 **검사 가능한 방어**다. 그리고 내가 아는 한 아무도 안 한다.

### 6.5 거리 계산에서 실측이 가르친 것

**사다리는 사용량 90%로 자른다.** Linear는 반경을 11종 선언하지만 90% 컷이면 7단이다.
디자이너가 실제로 이름 붙일 만한 수다. 그리고 `0px`은 간격 값의 **81%**라 반드시 뺀다.

**포함은 유사가 아니다.** 짧은 8pt 사다리는 긴 사다리 안에 구조적으로 들어간다.
그걸 유사도로 읽으면 8pt 그리드를 쓰는 모든 사이트가 서로의 복제가 된다.
Jaccard의 합집합 분모가 *공유하지 않는 칸*에 대해 벌점을 매기는 게 정확히 그 이유다.

**그리고 산술평균이 아니라 기하평균이다.** 이게 가장 늦게 발견됐다.

> 사다리가 완전히 달라도(0.385, 0.143) 나머지 다섯 성분이 우연히 일치하면
> **총점 0.632** — 복제로 오탐된다. 싼 성분들의 합의가 비싼 성분들의 불일치를 덮는다.
>
> **디자인의 정체성은 사다리에 있다.** 기하평균에서는 어느 한쪽이 0에 가까우면
> 곱이 끌려 내려가고, 부수적 일치로는 되살릴 수 없다.

고친 뒤 실측: 짝퉁 **0.990**, 무관한 8pt 페이지 **0.484**, 정반대 디자인 **0.022**.

### 6.7 세분화 — 보드는 세 층위다 (실전 런에서 배운 것)

첫 실전 런의 scout는 소스 3개를 물어왔다. 그건 스카우팅이 아니라 유명한 이름 대기다.
이제 최소 8캡처: 전체 페이지 2–3(느낌), 컴포넌트 3–5(`--selector`, 해부학),
**타이포그래피 스터디 1+**, **모션 스터디 1+**, 렌더 불가능한 것은 `--image`(추론만).

### 6.8 타이포와 모션도 측정한다

그 런이 배송한 "세리프 제목 + 기본 산세리프 다크 블로그"는 스페이싱·반경 사다리로는
안 잡힌다. 생성물의 가장 큰 tell — **선택하지 않아서 생긴 폰트** — 이 측정 밖이었다.

불변량에 여섯이 추가됐다: `typeScale` · `fontFamilies` · `weightLadder` ·
`motionDurations` · `easingVocab` · `animatedShare`. 이제 "멋있는 애니메이션"은
누가 실제로 읽은 숫자(대개 100–200ms + 진짜 커브)로 해석되고, 거리 계산도 타이포를
베낀 것과 아닌 것을 구분한다.

### 6.9 핑크 엘리펀트 — 프레임 누출은 결정론적으로 잡힌다

실전 런은 frame.md의 재프레이밍 문장을 **그대로 히어로에 박았다.** "하지 마"라고
하면 "안 합니다"를 쓰는 실패의 프레임판이다. 이건 측정 가능하다:

> **SLOP-LEAKED-RATIONALE** — 페이지 텍스트와 `.omd/frame.md`·`decisions.md` 사이에
> 연속 5토큰 창이 겹치면 위반. 설계 근거는 주방에 있고, 접시에 오르면 안 된다.

부정형 메타 카피("이런 내용은 없습니다", "We don't use jargon")와 im-not-ai 계열
한국어 AI 문체 틱(결론적으로, 첫째/둘째, ~를 통해…)은 SLOP-COPY가 확장해 잡고,
나머지 산문 규율은 hand 프롬프트가 진다 — 규칙이 못 재는 것은 지시가 진다.

### 6.6 레퍼런스는 어디서 오는가

| 출처 | 방식 | 상태 |
| :-- | :-- | :-- |
| 사용자가 지목한 공개 URL | 우리 추출기로 직접 렌더·측정 | ✅ 동작 확인 (Linear) |
| Mobbin | 공식 MCP/API (크레딧 기반) | 🔍 검토 |
| Dribbble | 공식 API | 🔍 검토 |
| 스크래핑 | — | ❌ **하지 않는다** |

**법적 경계를 명확히 한다.** Mobbin 약관은 서비스의 재현·복제를 금지하고,
Dribbble은 경쟁 서비스 구축을 금지한다. 우리는 **재배포하지 않는다.**

- 저장하는 것은 **이미지가 아니라 측정값과 원리 문장**이다.
- 저장 위치는 **사용자의 로컬 레포**다. 우리 서버는 없다.
- 공개된 페이지를 브라우저로 렌더해 **사실(간격, 반경, 대비비)을 측정**한다.
  사실은 저작물이 아니다.

> 그래도 회색지대다. 상용화 전에 법률 검토가 필요하다. **리스크 표에 올려둔다.**

## 7. 루프

```text
        ┌──────────────────────────────────────────────────────┐
        │                                                      │
        ▼                                                      │
   ① FRAME ─▶ ② CONCEPT ─▶ ③ REFERENCE ─▶ ④ COMMIT ─▶ ⑤ BUILD ─┤
   문제 의심    컨셉 선택      실물 관찰     구조 하나,    하나만
        ▲                                  비용 명시      제대로
        │                                                      │
        └───────────────── ⑦ REFRAME ◀──────────── ⑥ SEE ──────┘
                    "본 것이 문제를 바꾼다"        렌더·계측

                              ⑧ STOP
```

**⑦이 이 제품의 전부다.** 어떤 도구도 자기가 방금 만든 것을 스스로 폐기하고
문제 정의로 돌아가지 않는다. 그게 사람이 하는 일이다.

### 7.1 폐기된 설계: 후보 생성

이 루프는 두 번에 걸쳐 "여러 안을 만들고 고르는" 단계를 가졌다가 **전부 폐기했다.**

처음엔 `omd-hand` ×3이 완성본 셋을 병렬로 지었다. 버릴 것에 마감을 넣는 낭비라서
스케치 셋 + 완성 하나로 줄였다. 그런데 스케치도 같은 결함을 갖고 있었다:

> **베팅 셋을 오케스트레이터가 한 컨텍스트에서 정해서 넘긴다.**
> 격리가 막으려던 앵커링은 격리 *이전에* 이미 일어난다.
> 서브에이전트 셋은 아무것도 방어하지 않고 토큰만 3배 쓴다.
> 발산은 측정되지 않는 의식이었다.

지금은 **구조 하나에 커밋하고 그 비용을 문장으로 명시**한다. 평균으로의 수렴은
슬롭 린터가 *측정*으로 막고, 구조가 틀렸으면 REFRAME이 재건축한다. 재건축 한 번이
매번 후보 셋을 만드는 것보다 싸다.

> 대가 하나를 정직하게: 선택 단계가 사라지면서 **Layer 3(취향)의 자동 수집원도
> 사라졌다.** `omd choose`는 CLI에 남아 있지만 루프가 더는 후보를 제조해 먹이지
> 않는다. 취향은 이제 사용자가 결과물을 고치거나 기각할 때만 쌓인다. 그게 더
> 정직한 신호이기도 하다 — 만들어진 A/B보다 실제 교정이 진짜 선호다.

### 7.2 프롬프트로 쓴 루프는 반드시 무너진다

SKILL.md에 글로 적어주면 실제로는 이렇게 동작한다.

| 지시 | 모델의 실제 행동 | 원인 |
| :-- | :-- | :-- |
| 문제를 의심하라 | 한 문장 의심하는 척하고 넘어감 | 사용자를 만족시키려는 압력 |
| 레퍼런스를 참고하라 | 첫 번째 걸 그대로 베낌 | **Design fixation** |
| 구조에 커밋하라 | 처음 떠오른 것에 커밋 = 학습 분포의 평균 | 비용을 명시하게 해 한 박자 세운다 |
| 렌더해서 봐라 | 코드를 보고 "잘 보입니다" | 렌더 도구가 없으면 상상한다 |
| 재정의하라 | 방금 만든 걸 부정하기 싫어함 | 자기비평 맹점 |

**루프를 프롬프트에 쓰면 루프가 아니라 루프에 대한 묘사가 된다.**

### 7.3 그래서 구조로 만든다

```text
CLI            결정론적 계측          → "상상함"을 막는다
               대비비는 함수가 계산한다. 슬롭은 숫자로 잡힌다.

레퍼런스 IR     픽셀이 아니라 측정값    → "고착"을 막는다
               모델은 그림을 못 본다. 불변량과 원리 문장만 본다.
               omd ref distance 가 복제를 사후 검출한다.

서브에이전트    컨텍스트 격리          → "앵커링·자기비평 맹점"을 막는다
               sketch ×3이 서로를 못 본다 ⇒ 진짜 대안이 나온다. (싸다)
               eye는 hand의 의도를 모른다 ⇒ 진짜 비평을 한다.

.omd/          기록                  → "재정의 회피"를 막는다
               frame.md는 덮어쓰지 않고 덧붙인다.
```

### 7.4 강제하지 않는다

한때 프레임 승인 전까지 파일 쓰기를 훅으로 막았다. **지웠다.**

틀린 재프레이밍은 **만들어보고, 렌더해서 보고, 되돌아가는 것**으로만 잡힌다.
그런데 게이트가 서 있으면 승인 전엔 아무것도 못 만들고 → 볼 것이 없고 →
**재프레이밍이 틀렸는지 알 방법이 없다.** 루프를 그려놓고 그 루프를 막고 있었다.

> **원칙: 관찰로 검증될 일을 강제로 대신하지 않는다.**
> 그리고 사용자에게 터미널에서 명령어를 치라고 시키지 않는다.
> 그들이 이 도구를 켠 이유가 그 일을 하기 싫어서다.

### Phase 4 · Coach — 완료

```bash
omd coach
```

`decisions.md`와 `preferences.jsonl`을 읽어 **왜 그 수가 더 좋았는지** 설명하고,
반복되는 약점을 짚고, 다음에 볼 것을 준다.

> *"당신은 3개월간 대비비 지적을 41회 받았습니다. 1분기 대비 -68%.
> 대신 정보 계층 지적이 12→29회로 늘었습니다."*

**이게 "대체가 아니라 증폭"의 유일한 증거다.** 없으면 우리는 그냥 더 나은 v0다.

> "촘촘한 레이아웃을 좋아한다"는 취향이고 옳고 그름이 없다(α=0.248).
> "대비비를 반복해서 놓친다"는 실력이고 옳고 그름이 있다.
> **취향은 배우고, 실력은 가르친다.**

| 레퍼런스 수집의 법적 경계 | 🟠 중상 | 재배포 없음 · 측정값만 · 로컬 저장. **법률 검토 필요** |
| FRAMER가 잘난 척으로 읽힘 | 🟠 중상 | 한 문장으로 지나가듯. `--why` 없으면 CLI가 거부 |
| 임계값 취약성 | 🟠 중상 | hue 밴드 사례(§5.2). **사례 없이 조정 금지** |
| Coach 없이는 "증폭"이 구호에 그침 | 🟠 중상 | Phase 4 |
| 디자이너가 CLI를 안 씀 | 🟡 중 | 1차 타겟은 **디자인 겸업 개발자** |

**첫 두 줄이 제품의 생사다.**
레퍼런스가 짝퉁을 낳으면 슬롭을 지운 자리에 다른 슬롭이 들어온다.
슬롭 탐지가 Linear를 슬롭이라 부르면 아무도 이 도구를 켜지 않는다.

## 12. 한 문장으로

> 사람처럼 디자인한다는 건 좋은 화면을 뽑는 게 아니다.
> **문제를 의심하고, 컨셉을 세우고, 실물을 관찰해 원리를 뽑고,
> 싸게 여럿 그려보고 하나만 완성하고, 보고, 문제 정의를 갈아엎는 것**이다.
>
> "좋다"에 보편 함수가 없으므로(α = 0.248),
> 우리가 할 수 있는 건 **평균을 탐지하고, 실물에서 원리를 배우고, 당신의 취향을 익히는 것**이다.

**레퍼런스는 베끼는 대상이 아니라 관찰 대상이다.**
그래서 모델에게 그림을 주지 않는다. **숫자와 이유를 준다.**
