Why the connection pool empties itself during a traffic burst

08 Jul 2026 · 11 min read

Last month our checkout service started returning 502s in bursts of thirty to sixty seconds, always during traffic spikes, always recovering on its own before anyone could attach a debugger. PgBouncer's connection count looked fine. Postgres itself wasn't under load. The service metrics showed request latency climbing to the timeout ceiling and then falling back to normal, over and over, for two days before we understood why.

The short version: a retry loop was quietly doubling our connection demand every time the database got slow, and the database only got slow because the retry loop had doubled demand. It's a feedback loop with a one-line cause, and it's worth walking through because the same shape shows up in message queues, HTTP clients, and job schedulers.

The setup

We run a pool of 40 connections per service instance, backed by PgBouncer in transaction mode, backed by a Postgres primary with room for roughly 400 concurrent backends. Under normal load each instance holds 8 to 12 connections open at a time. That leaves headroom — or so the sizing spreadsheet said.

The client code wrapped every query in a retry helper, written two years ago by someone who has since left the team:

async def run_query(pool, sql, params, attempts=3):
    for attempt in range(attempts):
        try:
            async with pool.acquire() as conn:
                return await conn.fetch(sql, *params)
        except asyncpg.TooManyConnectionsError:
            await asyncio.sleep(0.05)
            continue
    raise PoolExhausted(sql)

Nothing here is obviously wrong. A connection request fails, we back off fifty milliseconds, we try again. The trouble is what "try again" means when the pool is the acquire call itself, not a separate retry client. pool.acquire() doesn't return an error when the pool is full — it blocks. The except clause was catching an error that could only come from PgBouncer refusing a connection outright, which happens under one specific condition: PgBouncer's own queue is full.

What actually happened under load

During a burst, a handful of slow queries — nothing pathological, just a join that took 400ms instead of 40ms — held connections slightly longer than usual. Requests queued behind them. PgBouncer's default queue depth is generous, so it absorbed the first wave without complaint. But our retry helper didn't wait for the queue; every blocked acquire() call was itself a task sitting in the event loop, and the moment PgBouncer's queue filled even briefly, the resulting error triggered a retry from every task that was queued at that instant.

Each retry was a fresh acquire() call, which went back into the same queue it had just failed to get through — except now there were two attempts in flight for every original request. The failure signal that was supposed to relieve pressure was manufacturing more of it.

The pool size and the retry count were both reasonable numbers on their own. Multiplied together during a queue-full event, they produced roughly 2.3x the intended connection demand within four seconds.

The fix

We replaced the blind retry with a bounded semaphore that tracks in-flight acquisition attempts separately from completed ones, and we made the backoff proportional to how full the queue already was rather than a flat fifty milliseconds:

async def run_query(pool, sql, params, attempts=3):
    async with inflight_limiter:
        for attempt in range(attempts):
            try:
                async with pool.acquire(timeout=2.0) as conn:
                    return await conn.fetch(sql, *params)
            except asyncpg.TooManyConnectionsError:
                backoff = 0.05 * (2 ** attempt)
                await asyncio.sleep(backoff)
        raise PoolExhausted(sql)  # surfaces as a 503, not a hang

The inflight_limiter caps total concurrent acquisition attempts at 1.5x the pool size, which means retries now compete with each other for a bounded slot instead of multiplying freely. Combined with exponential backoff, a queue-full event now drains in under a second instead of compounding for a minute.

What we'd check first next time

  • Any retry wrapped around a resource acquisition call, not just an RPC — acquisition failures behave differently from request failures and deserve their own backoff policy.
  • Whether the thing being retried is itself queue-based. A queue that's temporarily full is not the same failure mode as a queue that's permanently broken, and treating them the same is where the amplification comes from.
  • Whether retry attempts are bounded in aggregate across the process, not just per call site. Forty concurrent requests each retrying three times is a hundred and twenty attempts, not forty.

We added an alert on PgBouncer's queue depth crossing zero for more than one second, which is now our earliest signal — it fires a full minute before request latency moves.
