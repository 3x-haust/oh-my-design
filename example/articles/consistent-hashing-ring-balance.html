Consistent hashing rings are never as balanced as the diagram

22 Jun 2026 · 9 min read

Every introduction to consistent hashing draws the same picture: a circle, a handful of nodes placed evenly around it, keys falling into arcs of roughly equal size. It's a clean diagram and it's a lie about how the ring behaves with a small number of physical nodes. We ran a five-node cache cluster for over a year before checking the actual key distribution, and one node was carrying 31% of traffic while another carried 9%.

Where the imbalance comes from

With five points scattered on a ring by a single hash per node, the gaps between them are whatever the hash function happens to produce — there's no mechanism enforcing even spacing. Add enough nodes and the law of large numbers smooths this out, but five is nowhere near enough. The standard fix is virtual nodes: instead of hashing each physical node once, you hash it a hundred or two hundred times under different labels and place all of those points on the ring. More points means the gaps average out.

class HashRing:
    def __init__(self, nodes, vnodes=150):
        self.ring = {}
        self.sorted_keys = []
        for node in nodes:
            for i in range(vnodes):
                key = self._hash(f"{node}#{i}")
                self.ring[key] = node
        self.sorted_keys = sorted(self.ring)

    def _hash(self, s):
        return int(hashlib.md5(s.encode()).hexdigest(), 16)

    def get_node(self, key):
        h = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]

This is close to what we had. 150 virtual nodes per physical node, MD5 for the hash, sorted keys with a binary search for lookup. On paper, 150 vnodes should bring the standard deviation of load across nodes down to a few percent. Ours was nowhere close, and the code above isn't wrong — the input to it was.

The actual bug: key naming, not the ring

Our vnode labels were f"{node}#{i}", and our node names were hostnames like cache-a1, cache-a2, cache-b1. Four of the five hostnames were structurally similar strings differing by one or two characters at a fixed position. MD5 doesn't care about that in theory — a good hash scrambles similar inputs into unrelated outputs. But we weren't hashing the hostname alone; a config templating step had prefixed every label with the same sixteen-character environment string before it reached the ring constructor, and that shared prefix meant the effective entropy going into MD5 was smaller than it looked.

The tell was in the vnode placement itself, not the hash function's reputation. When we dumped the sorted ring positions, four nodes' virtual points clustered into recognizable bands instead of scattering uniformly — a symptom of low-entropy input, not a broken hash.

# before: shared prefix compresses effective entropy
label = f"{env_prefix}-{node}#{i}"

# after: hash the variable part first, mix in the rest
label = hashlib.sha256(f"{node}#{i}".encode()).hexdigest()

Switching to SHA-256 over just the node name and vnode index, with the environment prefix dropped from the hash input entirely, fixed the clustering. Standard deviation across the five nodes dropped from around 9 percentage points to under 2 within a day of rollout — no change to the vnode count, no change to the lookup algorithm, just different bytes going into the hash.

What we check now

  • Plot actual vnode positions on the ring before trusting the theoretical balance guarantee. A histogram of gap sizes takes ten minutes to build and tells you immediately whether the input entropy is doing its job.
  • Watch for any templating or namespacing layer between the "logical" node name and what actually gets hashed — a shared prefix or suffix is invisible in code review but visible in the distribution.
  • Increasing vnode count is not a fix for low-entropy labels. We tried 300 vnodes before finding the real cause; it made the imbalance marginally smaller and cost us a slower rebalance on every node add or remove.

The ring algorithm was fine the whole time. The lesson that stuck was to distrust the diagram's promise of "roughly equal arcs" until we'd actually measured the arcs.
